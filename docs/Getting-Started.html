1 Getting Started

Warning: All documentation and code examples are of how I hope this will ultimately work. Right now, most of this is not implemented.

To get things started, we will try to run a very simple GTK+ based GUI application using the bindings provided by Guile GI.

Now, to understand how this works, one needs to distinguish between what Guile-GI provides and the bindings that it creates.

  • Guile-GI itself is a library that allows you to parse typelib files, which describe code modules. It has a handful of procedures for inspecting typelib files as well as some helper procedures for dynamically creating procedures and types.
  • When Guile-GI is used, it creates procedures, variables, and types according to the instructions in the typelib files. These are created at runtime when the Guile-GI procedure typelib-load is called.

First, for each GObject library that you want to use in your program, you need to have an export library. A sample library for GTK+3 could take the following form. 

(define-module (gi gtk-3)
  #:use-module (gi))

(typelib-load "Gtk" "3.0")

Note that Guile-GI provides export libraries for the most common GObject libraries. It includes (gi gtk-3), (gi gio-2), and (gi glib-2), for example.

Alternatively, you can instruct Guile-GI to create these modules for you, as shown in the following. 

(use-modules (gi))
(use-typelibs ("GLib" "2.0") ("Gtk" "3.0"))

Here, Guile-GI creates the (hopefully unique) modules (%gi GLib-2.0) and (%gi Gtk-3.0) and imports them into the current module using (use-modules).

For these auto-generated libraries, you are going to need some documentation. The procedure typelib-document creates some rough documentation for a typelib: it gives the procedure names and how they are called, plus a list of types and constants. For example, the following commands would make some rough documentation for GTK and GIO. 

guile -L . -c '(use-modules (gi)) (display (typelib-document "Gtk" "3.0"))' > Gtk.txt
guile -L . -c '(use-modules (gi)) (display (typelib-document "Gio" "2.0"))' > Gio.txt

See Typelib Introspection for more information about typelib-load and typelib-document.

A minimal GTK application looks like this 

(use-modules (gi)
             (gi gio-2)
             (gi gtk-3)
             (gi glib-2))

(define (print-hello widget)
  (display "Hello World\n"))

(define (activate-callback app)
  (let* ((window (create <GtkApplicationWindow>
                   (application app)
                   (default-height 200)
                   (default-width 200)
                   (title "Window")))
         (button-box (create <GtkButtonBox> (parent window)))
         (button (create <GtkButton>
                   (parent button-box)
                   (label "Hello world"))))
    (with-object button
      (connect! clicked print-hello)
      (connect! clicked (lambda _ (with-object window (destroy)))))

    (with-object window (show-all))))

(define (main)
  (with-object (create <GtkApplication> (application-id "org.gtk.example"))
    (connect! activate activate-callback)
    (run (length (command-line)) (command-line))))

(main)

When it is run, it will create an empty window with a title. Are you impressed?

Let’s walk through this example. First off, you may notice two important pieces of syntax: create and with-object. These three macros allow one to interact with GObject’s very odd class system.

First is create, which – as the name implies – creates an object. The first argument is a type, which can either be a “proper” type, such as <GtkApplication> or a number1. This will be the type of the resulting object. The rest of the arguments are tuples (key value), where key is the name of a field and value is a value to store therein. These pairs are passed to the object at creation time as strings and GValues respectively. Thus it is possible to set CREATE_ONLY properties with it.

Next is the with-object syntax. Within its scope function calls will be translated to method calls of the given object. Hence with-object window (show-all) translates roughly to gtk_widget_show_all(window). As can be seen in this example, it matters little if a method is defined in the object’s class or one of its parent – it is resolved either way.

And lastly is the modify-signals syntax. It hooks procedures to GObject signals. Signals are its native way of advising objects of the occurrence of events.

Previously, we’ve used send to invoke functions and connect to hook signals, which shadowed Guile procedures for the use with sockets. This was done to conform to the official GObject Introspection Guidelines, which are as follows.

Try to stay close to the C language semantics, for instance GObject should be mapped to a class and gtk_button_set_label to a method of that class.

  1. ...
  2. scheme: (send button (set-label "foo"))

Shadowing Guile procedures is certainly not ideal. A workaround would have been to rename these procedures as user, but this does away with the supposed benefits of their names, on top of being a somewhat pointless typing exercise. If we already have to break either specification, we want to do it in a way, that makes sense. Hence we introduce with-object, which acts as a safe barrier (both visually and effectively) around GObject methods, that would otherwise cause such confusion.

See GObject Signals for more information on modify-signals. See with-object syntax for information on the with-object syntax.
